流程
1、解析用户需求 并展示供确认
2、禁止猜测 任何不明确必须发起提问
多个不明确需要逐个提问 禁止同时发起多个提问
收集回复信息后 重复执行步骤1
3、提问方式使用菜单项进行回答
4、编码之前必须展示完整方案 用户未最终确认 不得执行
5、必须在任务开始 进行 结束 的时候展示使用的skill

以下规则必须遵守
try-with-resources
禁止任何魔法值 全部采用枚举
非必要不得使用Map等 尽可能使用类进行封装
比如接口入参 出参 禁止使用map等弱类型
状态码采用HTTP STATUS
最小改动 用最少的代码量和影响范围实现功能
注释 采用javadoc标准 注释必须精炼直白 逻辑要写明步骤 关键步骤
日志 采用log4j2 关键步骤必须进行日志打印 关键信息需要脱敏
代码中使用短返回 避免深层次嵌套
不得调用同层的bean 比如service层不得调用其他的service 统一使用controller进行调用
可以复用的逻辑 允许抽取到组合service中注入多个service进行逻辑编排
参数校验统一使用@Valid单个校验 @Validation分组校验
使用池化 如数据库连接池 异步任务线程池
线程池优化参数 用户不填写则自动加载系统资源参数进行自动调优
异步任务 提交线程池执行
类转换应该使用充血模型 避免膨胀
使用Localdatetime替换Date等
通常时间格式采用 yyyy-MM-dd HH:mm:ss
查询数据是否存在 使用EXISTS
查询数据库仅获取必要的字段 尽可能避免 SELECT *
多表查询尽可能简化 禁用子查询 考虑JOIN优化查询结构
数据改动逻辑需要使用事务包裹 事务包裹的范围尽可能小
阶段任务完成后
需要编译 不通过需要解决
检查功能是否正常
对比需求分析是否达标
循环依赖需要解耦
代码生成顺序参考 实体类=>Mapper=>Controller=>service
ORM采用Mybatis+MybatisPlus 
Service需要继承IServcie
Impl需要继承ServiceImpl
Mapper需要继承BaseMapper 
组合服务属于高自定义 所以无需继承MP
分页参数使用header进行传递 pageSize pageNum
分页逻辑需要考虑深分页优化
分页参数获取通过注入获取
数据库表设计
表主键使用雪花算法
必须具有审计字段 id create_by create_time update_by update_time deleted
字段尽可能使用 NOT NULL DEFAULT
根据业务考虑索引设计
