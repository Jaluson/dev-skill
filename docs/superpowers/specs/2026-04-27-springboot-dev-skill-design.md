# Spring Boot 开发规范 Skill 设计文档

## 1. 概述

将 `springboot开发规范草稿.md` 转化为可执行的 Claude Skill，提供**开发模式**（编码时遵循规范）和**审查模式**（检查代码合规性）两种工作方式。

## 2. 目标

- 开发模式：在生成 Spring Boot 代码时，自动应用规范约束
- 审查模式：对已有代码进行合规性扫描，输出问题列表与修复建议
- 统一规范来源：所有规则源自同一份规范草稿，避免开发与审查标准不一致

## 3. Skill 定义

| 属性 | 值 |
|------|-----|
| 名称 | `springboot-dev` |
| 触发命令 | `/springboot-dev` |
| 参数 | `[--review] [--full \| --diff]` |
| 默认行为 | 开发模式 |

### 3.1 参数说明

| 参数 | 含义 | 默认 |
|------|------|------|
| 无参数 | 进入开发模式 | 是 |
| `--review` | 进入审查模式 | 否 |
| `--full` | 全量审查（整个项目） | 审查模式下默认 |
| `--diff` | 增量审查（仅变更文件） | 需显式指定 |

## 4. 双模式架构

```
┌─────────────────────────────────────────┐
│           /springboot-dev               │
│  ┌─────────────┐    ┌─────────────┐    │
│  │  开发模式    │    │  审查模式    │    │
│  │  (默认)      │    │  (--review) │    │
│  └──────┬──────┘    └──────┬──────┘    │
│         │                  │            │
│         ▼                  ▼            │
│  ┌─────────────┐    ┌─────────────┐    │
│  │ 需求解析    │    │ 扫描代码    │    │
│  │ → 确认     │    │ → 检查规则   │    │
│  │ → 方案     │    │ → 输出报告   │    │
│  │ → 编码     │    │             │    │
│  └─────────────┘    └─────────────┘    │
│         │                  │            │
│         └────────┬─────────┘            │
│                  ▼                      │
│         ┌─────────────┐                │
│         │ 规范知识库   │                │
│         │ (统一来源)   │                │
│         └─────────────┘                │
└─────────────────────────────────────────┘
```

两种模式共享同一套规范知识库，确保标准一致。

## 5. 规范模块映射

将原始规范拆分为 7 个独立模块，skill 根据当前场景自动加载相关模块。

### 5.1 流程规范（仅开发模式）

| 规则 | 说明 |
|------|------|
| R-FLOW-01 | 解析用户需求后必须展示供确认 |
| R-FLOW-02 | 任何不明确点必须发起提问，禁止猜测 |
| R-FLOW-03 | 多个不明确点需逐个提问，禁止同时发起多个提问 |
| R-FLOW-04 | 收集回复后重复执行确认步骤 |
| R-FLOW-05 | 提问方式使用菜单项进行回答 |
| R-FLOW-06 | 编码前必须展示完整方案，用户未确认不得执行 |
| R-FLOW-07 | 任务开始、进行、结束时展示使用的 skill |
| R-FLOW-08 | 阶段任务完成后必须编译，不通过需解决 |
| R-FLOW-09 | 检查功能是否正常 |
| R-FLOW-10 | 对比需求分析是否达标 |

### 5.2 编码规范（双模式）

| 规则 | 说明 | 审查检查方式 |
|------|------|-------------|
| R-CODE-01 | 必须使用 try-with-resources | 语法扫描 |
| R-CODE-02 | 禁止魔法值，全部采用枚举 | 常量/字符串字面量扫描 |
| R-CODE-03 | 非必要不得使用 Map，入参出参必须使用类封装 | 方法签名扫描 |
| R-CODE-04 | 状态码采用 HTTP STATUS | 响应类扫描 |
| R-CODE-05 | 最小改动原则 |  diff 范围扫描 |
| R-CODE-06 | 注释采用 Javadoc 标准，精炼直白，逻辑写明步骤 | 注释扫描 |
| R-CODE-07 | 日志采用 log4j2，关键步骤必须打印，关键信息脱敏 | 日志语句扫描 |
| R-CODE-08 | 使用短返回，避免深层次嵌套 | 代码结构扫描 |
| R-CODE-09 | 不得调用同层 bean（如 Service 不得调用其他 Service） | 依赖注入扫描 |
| R-CODE-10 | 可复用逻辑允许抽取到组合 Service 中编排多个 Service | 类结构扫描 |
| R-CODE-11 | 参数校验统一使用 @Valid（单校验）或 @Validated（分组校验） | 方法参数扫描 |
| R-CODE-12 | 使用 LocalDateTime 替代 Date | 类型扫描 |
| R-CODE-13 | 时间格式采用 `yyyy-MM-dd HH:mm:ss` | 格式化字符串扫描 |
| R-CODE-14 | 类转换使用充血模型，避免膨胀 | 类结构扫描 |

### 5.3 架构规范（双模式）

| 规则 | 说明 | 审查检查方式 |
|------|------|-------------|
| R-ARCH-01 | ORM 采用 MyBatis + MyBatisPlus | 依赖扫描 |
| R-ARCH-02 | Service 继承 `IService<T>` | 类继承扫描 |
| R-ARCH-03 | Impl 继承 `ServiceImpl<Mapper, T>` | 类继承扫描 |
| R-ARCH-04 | Mapper 继承 `BaseMapper<T>` | 类继承扫描 |
| R-ARCH-05 | 组合 Service 属于高自定义，无需继承 MP | 类结构扫描 |
| R-ARCH-06 | 循环依赖需要解耦 | 依赖图扫描 |

### 5.4 数据库规范（双模式）

| 规则 | 说明 | 审查检查方式 |
|------|------|-------------|
| R-DB-01 | 表主键使用雪花算法 | 实体类/表结构扫描 |
| R-DB-02 | 必须具有审计字段：`id`, `create_by`, `create_time`, `update_by`, `update_time`, `deleted` | 实体类/表结构扫描 |
| R-DB-03 | 字段尽可能使用 `NOT NULL DEFAULT` | 表结构扫描 |
| R-DB-04 | 根据业务考虑索引设计 | 表结构扫描 |
| R-DB-05 | 查询数据是否存在使用 `EXISTS` | SQL 扫描 |
| R-DB-06 | 查询仅获取必要字段，避免 `SELECT *` | SQL 扫描 |
| R-DB-07 | 多表查询尽可能简化，禁用子查询，考虑 JOIN 优化 | SQL 扫描 |
| R-DB-08 | 数据改动逻辑使用事务包裹，范围尽可能小 | 方法注解扫描 |

### 5.5 ORM / 分页规范（双模式）

| 规则 | 说明 | 审查检查方式 |
|------|------|-------------|
| R-ORM-01 | 代码生成顺序：实体类 → Mapper → Controller → Service | 项目结构扫描 |
| R-ORM-02 | 分页参数使用 header 传递：`pageSize`, `pageNum` | 接口扫描 |
| R-ORM-03 | 分页逻辑考虑深分页优化 | 代码逻辑扫描 |
| R-ORM-04 | 分页参数通过注入获取 | 参数获取方式扫描 |

### 5.6 线程池与资源规范（双模式）

| 规则 | 说明 | 审查检查方式 |
|------|------|-------------|
| R-POOL-01 | 使用池化：数据库连接池、异步任务线程池 | 配置扫描 |
| R-POOL-02 | 线程池参数用户不填写则自动加载系统资源调优 | 配置扫描 |
| R-POOL-03 | 异步任务提交线程池执行 | 代码扫描 |

### 5.7 文档规范（开发模式生成，审查模式检查）

| 规则 | 说明 |
|------|------|
| R-DOC-01 | 输出文档命名：`yyyy-MM-dd_HH:mm:ss-[模块]-[功能]-[文档类型].md` |
| R-DOC-02 | 目录结构：`后端文档/接口文档`、`接口改动记录`、`SQL文档/DDL`、`SQL文档/数据库改动记录`、`SQL文档/DML`、`curl测试文档`、`操作记录`、`功能改动记录` |
| R-DOC-03 | 新增/改动接口必须保持最新接口文档 |
| R-DOC-04 | 数据库改动需记录 DDL 和 DML |

### 5.8 环境规范（开发模式）

| 规则 | 说明 |
|------|------|
| R-ENV-01 | 环境变量路径：`项目根目录/project.env.yaml` |
| R-ENV-02 | 使用 YAML 格式配置 |
| R-ENV-03 | 文件不存在时先创建，根据项目内配置文件判断所需环境 |

## 6. 开发模式工作流程

```
1. 用户输入需求
   → Skill 解析并结构化展示
   → 明确点 / 不明确点标注

2. 存在不明确点？
   → 是：逐个提问（菜单项方式），回到步骤 1
   → 否：继续

3. 展示完整方案
   - 接口设计（URL / Method / 入参 / 出参）
   - 数据库设计（表结构 / 字段 / 索引）
   - 代码结构（类图 / 调用链）

4. 用户确认方案？
   → 否：回到步骤 1 或 3 修改
   → 是：继续

5. 执行编码
   - 按 R-ORM-01 顺序生成：实体类 → Mapper → Controller → Service
   - 每步应用对应编码/架构/数据库规范

6. 阶段完成后验证
   - 编译通过？
   - 功能正常？
   - 对比需求达标？
   → 否：修复后重新验证
   → 是：继续

7. 生成/更新文档
   - 接口文档
   - SQL 文档（DDL / DML）
   - curl 测试文档
   - 功能改动记录

8. 结束，展示使用的 skill
```

## 7. 审查模式工作流程

### 7.1 增量审查（`--review --diff`）

```
1. 获取变更文件列表（git diff --name-only）
2. 过滤出 Java / XML / YAML / SQL 文件
3. 对每个文件应用规则扫描
   - Java 文件：R-CODE, R-ARCH, R-DB(实体类), R-ORM, R-POOL
   - XML 文件：R-DB(SQL), R-ORM
   - YAML/Properties：R-POOL, R-ENV
4. 汇总问题列表
5. 输出审查报告
```

### 7.2 全量审查（`--review --full`）

```
1. 扫描整个项目的 Java / XML / YAML / SQL 文件
2. 对每个文件应用规则扫描（同增量审查）
3. 额外扫描项目结构合规性
   - 目录结构是否符合 Maven/Gradle 标准
   - 代码生成顺序是否正确（实体类 → Mapper → Controller → Service）
   - 文档目录是否完整
4. 汇总问题列表
5. 输出审查报告
```

## 8. 审查输出格式

```markdown
# Spring Boot 规范审查报告

## 概览
- 审查文件数：{count}
- 问题总数：{total}
  - [严重] {critical} 个
  - [警告] {warning} 个
  - [建议] {suggestion} 个

## 问题详情

### [严重] {规则编号} - {规则简述}
- 文件：`{path}:{line}`
- 问题：{具体描述}
- 规范来源：{模块}.{规则}
- 修复建议：{具体代码或步骤}

### [警告] {规则编号} - {规则简述}
- 文件：`{path}:{line}`
- 问题：{具体描述}
- 修复建议：{具体代码或步骤}

## 规范覆盖检查

| 模块 | 检查项 | 状态 |
|------|--------|------|
| 流程规范 | - | 仅开发模式适用 |
| 编码规范 | 14 项 | 已检查 {n} 项 |
| 架构规范 | 6 项 | 已检查 {n} 项 |
| 数据库规范 | 8 项 | 已检查 {n} 项 |
| ORM/分页 | 4 项 | 已检查 {n} 项 |
| 线程池 | 3 项 | 已检查 {n} 项 |
| 文档规范 | 4 项 | 已检查 {n} 项 |
```

## 9. Skill 文件结构

```
skills/springboot-dev/
├── skill.md              # Skill 定义（触发方式、描述、参数）
├── README.md             # 使用说明
└── rules/
    ├── flow.md           # 流程规范（R-FLOW-01 ~ 10）
    ├── code.md           # 编码规范（R-CODE-01 ~ 14）
    ├── architecture.md   # 架构规范（R-ARCH-01 ~ 06）
    ├── database.md       # 数据库规范（R-DB-01 ~ 08）
    ├── orm.md            # ORM/分页规范（R-ORM-01 ~ 04）
    ├── pool.md           # 线程池规范（R-POOL-01 ~ 03）
    ├── document.md       # 文档规范（R-DOC-01 ~ 04）
    └── environment.md    # 环境规范（R-ENV-01 ~ 03）
```

## 10. 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 非 Spring Boot 项目调用 | 提示当前目录不包含 Spring Boot 特征（pom.xml 无 spring-boot 依赖），询问是否继续 |
| 审查时发现规范冲突 | 以原始 `springboot开发规范草稿.md` 为准，输出冲突点供人工判断 |
| 用户要求跳过某项规范 | 记录到本次会话上下文，后续不再检查该项，但报告需标注 |
| 规范未覆盖的新场景 | 遵循最小改动原则，开发模式下主动询问用户 |
